Getting Flow Info Find out what flows are available to run

Intro

Before you can run a flow on the platform, you need to have the flow identifier and flow version identifier for the flow you wish to run. We recommend, that you get the version information at runtime rather than hard coding it into your apps. You should therefore either store the name of the flow (which is very loose coupling as names are subject to change if the author decides to do this) or the flow identifier only. This means you are always getting the latest activated version of the flow. If you hard code the version number, you will always get the same version of that flow - which means you'll never consume any updates as changes are made over time.

Flow Runtime Sequence

When loading and running a flow, the sequence of calls is typically the following:

  1. Get the flow version identifier (and flow identifier for "name" references)
  2. Initialize the flow, passing in any initialization values and annotations
  3. Execute the flow and consume the response
    1. If the response requires user interaction, write the page layout to the screen for the user (indicated by an invoke type of "FORWARD")
    2. If the response is simply a status message for a background process, write the wait info to the screen (indicated by an invoke type of "WAIT")
    3. In either case, the calling application should poll our "ping" API to check for any changes to the flow state (e.g. a background process has completed or another user has changed the view state or executed another step)

Initializing

Once you have the full Flow Identifier object, you can initialize your flow, ready to start executing! Initialization allows you to do a few things:

  1. Create inputs: Inputs are related to Values in a flow. If the author of the flow has created Values that have an access setting of "Input" or "Input/Output", you can pass information into those values at initialization. These might be things like the Id of an Account record in salesforce.com that the flow relates to, or the phone number of the current caller.
  2. Create annotations: An annotation is a piece of data you want to attach to your flow, but is not actually passed in as an input. Often developers want to store additional information in the flow state that helps with integration or more complex configuration. Think of annotations as session state that is read/write for the entire duration of the flow execution.
  3. Set the execution mode: The "mode" allows you to run the flow in a couple of different debug configurations. This allows you to step through the flow as a user would see it, but also step through the flow element by element (even logical elements). It's very powerful for debugging - honestly - it's fantastic when flows get a little more complicated!

Flow Response Object

The flow response object properties are as follows:

Property Type Description
dateCreated String: Date/Time in JavaScript 1.8.5/ECMAScript 5th Edition format The date the flow was first created.
dateModified String: Date/Time in JavaScript 1.8.5/ECMAScript 5th Edition format The date the last modificiation was made to the flow.
userCreated String: GUID The unique identifier for the user who created the flow.
userModified String: GUID The unique identifier for the user who last modified the flow.
userOwner String: GUID The unique identifier for the user who owns this flow.
editingSessionId String: GUID This property is returned, but is always 'null' for run operations.
editingSessionTokenId String: GUID This property is returned, but is always 'null' for run operations.
id Object: Flow Identifier Object The complete unique identifier needed to initialize and run the flow.
developerName String The developer name for the flow. If you are referencing flows by name, this is the name you should use in your referencing.
developerSummary String The developer summary the author provided to give more information about the flow.
isActive Boolean Indicates if this flow version is the active version. In the case of run operations, this will always be true.
isDefault Boolean Indicates if this flow version is the default version. In the case of run operations, this will always be true.
startMapElementId String: GUID The unique identifier for the first element in the flow.
authorization Object: Authorization The Authorization object for this flow.

Flow Identifier Object

The flow identifier object properties are as follows:

Property Type Description
id String: GUID The unique identifier for the flow. This identifier does not change for the lifetime of the flow and is generated when the flow is first created.
versionId String: GUID The unique identifier for the specific version of a flow. This identifier changes any time a change is made to the flow - in structure or content.

Engine Value Object

The engine value object properties are as follows:

Property Type Description
id String: GUID (Optional) The unique identifier for the value in the flow being assigned. Using the id to reference the value ensures that your player is guaranteed to be assigning the correct value. If you reference a value by developerName, it is possible for the author to break integration points simply by changing the name of the value. You must provide an 'id' or a 'developerName'.
typeElementId String: GUID (Optional) The unique identifier for the object type of the value in the flow being assigned. Using the typeElementId ensures that your player is guaranteed to be referencing the correct object type. If you use the typeElementDeveloperName property, a change in the type name can break the integration. We ask that you specify the type as we may in future support type casting and we therefore want to know the type you're passing in so we can validate it is correct before mapping to the super type.
typeElementEntryId String: GUID The unique identifier for the object property of the value in the flow being assigned. Typed objects all have properties or "Type Element Entries" as they're called through the API. Using the typeElementEntryId allows you to assign a property in an object value as opposed to the whole value. As with the other identifier references, using the typeElementEntryId ensures that your player is guaranteed to be assigning the correct value. Using the typeElementEntryDeveloperName makes it possible to break the integration if the author changes the developerName of a property in the type.
typeElementDeveloperName String (Optional) The developer name for the object type of the value in the flow being assigned. See the typeElementId for more information.
typeElementEntryDeveloperName String The developer name for the object property of the value in the flow being assigned. See the typeElementEntryId for more information.
contentValue String The actual content value being assigned to the value in the flow. This property should be used for all "primitive", non-typed values in your flow. For example, if you are referencing a value called "First Name", this would be the value you actually want to assign to it: e.g. "Steve".
contentType String: ContentType The content type of the value you are passing into the flow. The content types are specified by the ContentType enumeration.
id Object: Object Data Object The actual object data being assigne to the value in the flow. If you are assigning a typed object, you should use this property to assign the value. The object data is provided as a list of objects regardless of whether you're assigning an object or a list. However, for objects you should only ever have one object in your array.

ContentType Enumeration

The ContentType enum contains the following values:

Value Typed Description
ContentString No A Java java.lang.String or C# System.String value.
ContentNumber No A Java java.lang.Double or C# System.Double value.
ContentBoolean No A Java java.lang.Boolean or C# System.Boolean value.
ContentPassword No A Java java.lang.String or C# System.String value. If you indicate a value is a password, we use additional value encryption on storage.
ContentDateTime No A Java java.lang.DataTime or C# System.DateTime. Using the REST API, the value should be provided in JavaScript 1.8.5/ECMAScript 5th Edition format.
ContentContent No A Java java.lang.String or C# System.String value. If you indicate a value is "Content", we typically expect that the content holds HTML formatted String content. The engine will also parse the contents of this String with any value references.
ContentObject Yes The value is Object Data based on the ManyWho format. This allows you to pass complex data into ManyWho for processing, including child objects and lists. Any Object Data being passed into the system also needs to have a Type as defined by your Type Elements.
ContentList Yes The value is a list of Object Data based on the ManyWho format. This allows you to pass lists of complex data into ManyWho for processing (of the same Type), including child objects and lists. Any Object Data being passed into the system also needs to have a Type as defined by your Type Elements.
