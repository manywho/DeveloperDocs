Add To Flow

In order to use a Type Element in your Flow, you must 'add' or 'import' the element to the Flow so our engine can keep track of it as a dependency. If the Type is associated with a Service Element, it will automatically be imported into your Flow when the Service Element is added. To add a reference to a specific Type Element object to your Flow, you can use the following API:

URL HTTP Verb JSON Request JSON Response
api/draw/1/element/flow/{flowId}/type/{id} POST N/A N/A

Where the parameters are as follows:

Property Type Description
flowId String: GUID The unique identifier for the Flow you wish to add the Type Element to.
id String: GUID The unique identifier for the Type Element to be added to the Flow.

You must also provide the draw API with the following header value:

Name Value
Authorization The authorization String returned from logging into the api/draw/1/authentication API.
POST https://flow.manywho.com/api/draw/1/element/flow/7ba7a6a1-d6ba-4806-8cc0-77be8504e86d/type/365c14c4-78d4-4c01-b84d-19024b3ea1ce

Query

To query for Type Element objects, you can use the following API:

URL HTTP Verb JSON Request JSON Response
api/draw/1/element/type?filter={filter} GET N/A Array: Type Element Response Objects

Where the parameters are as follows:

Property Type Description
filter String The filter can currently supports two possible options - search by exact developer name or search by partial developer name. To search for an exact developer name use: filter=developerName eq 'name of your element' To search for by partial developer name use: filter=substringof(developerName,'partial name of your element') This follows the conventions established in the OData standard, though we do not currently support the $ prefix.

You must also provide the draw API with the following header value:

Name Value
Authorization The authorization String returned from logging into the api/draw/1/authentication API.
GET https://flow.manywho.com/api/draw/1/element/type?filter=substringof(developerName,'Customer')

Delete

To delete a specific Type Element object, you can use the following API:

URL HTTP Verb JSON Request JSON Response
api/draw/1/element/type/{id} DELETE N/A N/A

Where the parameters are as follows:

Property Type Description
id String: GUID The unique identifier for the Type Element you wish to delete. This will not delete the element in previous versions - it will only delete the Type Element from builds of your Flow done from this point forward.

You must also provide the draw API with the following header value:

Name Value
Authorization The authorization String returned from logging into the api/draw/1/authentication API.
DELETE https://flow.manywho.com/api/draw/1/element/type/365c14c4-78d4-4c01-b84d-19024b3ea1ce

JSON Objects Object structures

Type Element Request Object

The type element request object properties are as follows:

Property Type Description
id String: GUID The unique identifier for the Type Element. The id should be null for "insert" requests and a valid identifier for "update" requests. This property is created by the service.
elementType String: ElementType The type of element this metadata represents. For Type Elements, the only valid value currently is: TYPE
developerName String The developer name for the Type Element. This is useful for keeping track of the Type Element in the modeling tool and API.
developerSummary String (Optional) The developer summary the author provided to give more information about the Type Element.
serviceElementId String: Guid The unique identifier for the service that installed this Type Element. This is not currently attached to any functionality, but will be part of the "uninstall" capabilities of Service Elements.
properties Array: Type Element Property Objects The list of Properties for this Type Element. A Property is very similar to a field in a table - it represents the structure of the data that will be stored in Values that use this Type.
bindings Array: Type Element Binding Objects The list of Bindings for the Type Element. A Binding holds the mapping of properties to tables and fields in an underlying Service Element. A binding is not required for a Type Element if there is no plan to save the data in an external data store. When creating a binding, you will need to have saved the initial Type Element first so you have the identifiers for the various Properties in the Type that need to be bound.
updateByName Boolean Indicates if the platform should attempt to find a Type Element with the same developer name as the one provided and match them up by name as opposed to 'id'. This is useful when creating scripts to create Flows - as you can use the developerName property as the reference as opposed to needing to know the ids of all created Elements.

Type Element Response Object

The Type Element has all of the same properties as the request object, but additionally the properties listed below:

Property Type Description
dateCreated String: Date/Time in JavaScript 1.8.5/ECMAScript 5th Edition format The date the Type Element was first created.
dateModified String: Date/Time in JavaScript 1.8.5/ECMAScript 5th Edition format The date the last modificiation was made to the Type Element.
userCreated String: GUID The unique identifier for the user who created the Type Element.
userModified String: GUID The unique identifier for the user who last modified the Type Element.
userOwner String: GUID The unique identifier for the user who owns this Type Element.

Type Element Property Object

The type element property object properties are as follows:

Property Type Description
id String: GUID The unique identifier for the type element property value. This property is created by the service.
developerName String The developer name for the type element property. This will be used to identify the property in API calls that use values of this Type.
contentType String: ContentType The type of content the type element property holds.
typeElementId String: GUID The unique identifier for the Type Element held in this type element property. This property should only be assigned if this type element property is a contentType of ContentObject or ContentList. The unique identifer should be for the object or list of objects that will be stored.
typeElementDeveloperName String This property is used to specify the developer name rather than the identifier of the associated Type Element (see typeElementId). This property is only used as part of the Service Element install. Do not use this property if you are creating a Type Element through the API manually - you must use the typeElementId.

Type Element Binding Object

The type element binding object properties are as follows:

Property Type Description
id String: GUID The unique identifier for the type element binding. This property is created by the service.
developerName String The developer name for the type element binding.
developerSummary String (Optional) The developer summary the author provided to give more information about the type element binding.
databaseTableName String The name of the database table that this binding should be applied. If no underlying table is used or the binding involves multiple backend tables, this should represent a unique name that will allow the Service Element implementation to identify how to store the object.
serviceElementId String: GUID The unique identifier for the Service that implements the mapping code to save objects and lists of this Type back to the data store.
propertyBindings Array: Type Element Property Binding Objects The list of type element property bindings. These are the individual mappings of type element properties to fields in the underlying data store implemented by the Service.

Type Element Property Binding Object

The type element property binding object properties are as follows:

Property Type Description
id String: GUID The unique identifier for the type element property binding. This property is created by the service.
databaseFieldName String The name of the database field in the table that this binding should be applied. If no underlying table is used, this should represent a unique name that will allow the Service Element implementation to identify how to store the object property.
typeElementPropertyId String: GUID The unique identifier for the type element property that this binding relates to.
typeElementPropertyDeveloperName String: GUID The developer name for the type element property that this binding relates to. This property is only used in the Service Element installation and is not supported as part of a manual Type creation.
databaseContentType String: GUID The content type of the database field in the table that this binding should be applied. For example, the content type in ManyWho may be ContentNumber, but in the underlying database, the actual field type is Decimal. This is an optional property that depends on the Service implementation.

API Enumerations Helpful values

ContentType Enumeration

The ContentType enum contains the following values:

Value Typed Description
ContentString No A Java java.lang.String or C# System.String value.
ContentNumber No A Java java.lang.Double or C# System.Double value.
ContentBoolean No A Java java.lang.Boolean or C# System.Boolean value.
ContentPassword No A Java java.lang.String or C# System.String value. If you indicate a value is a password, we use additional value encryption on storage.
ContentDateTime No A Java java.lang.DataTime or C# System.DateTime. Using the REST API, the value should be provided in JavaScript 1.8.5/ECMAScript 5th Edition format.
ContentContent No A Java java.lang.String or C# System.String value. If you indicate a value is "Content", we typically expect that the content holds HTML formatted String content. The engine will also parse the contents of this String with any value references.
ContentObject Yes The value is Object Data based on the ManyWho format. This allows you to pass complex data into ManyWho for processing, including child objects and lists. Any Object Data being passed into the system also needs to have a Type as defined by your Type Elements.
ContentList Yes The value is a list of Object Data based on the ManyWho format. This allows you to pass lists of complex data into ManyWho for processing (of the same Type), including child objects and lists. Any Object Data being passed into the system also needs to have a Type as defined by your Type Elements.
