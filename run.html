Run

Getting Flow Info Find out what flows are available to run

Intro

Before you can run a flow on the platform, you need to have the flow identifier and flow version identifier for the flow you wish to run. We recommend, that you get the version information at runtime rather than hard coding it into your apps. You should therefore either store the name of the flow (which is very loose coupling as names are subject to change if the author decides to do this) or the flow identifier only. This means you are always getting the latest activated version of the flow. If you hard code the version number, you will always get the same version of that flow - which means you'll never consume any updates as changes are made over time.

Flow Runtime Sequence

When loading and running a flow, the sequence of calls is typically the following:

  1. Get the flow version identifier (and flow identifier for "name" references)
  2. Initialize the flow, passing in any initialization values and annotations
  3. Execute the flow and consume the response
    1. If the response requires user interaction, write the page layout to the screen for the user (indicated by an invoke type of "FORWARD")
    2. If the response is simply a status message for a background process, write the wait info to the screen (indicated by an invoke type of "WAIT")
    3. In either case, the calling application should poll our "ping" API to check for any changes to the flow state (e.g. a background process has completed or another user has changed the view state or executed another step)

Authenticating Users Getting access to a flow as an end user

Authentication as an end user has a couple of layers. This is because flows can be constructed to leverage multiple identity services - but also because we don't always deny the user complete access if they are not authorized. It works like this:

  1. Flow authentication: When a flow is created, the author must define the overall authentication for the flow - and the directory the user must authenticate against to gain access. The authentication process can rely on specific user access, group access or location requirements. If the user cannot authenticated at the flow level, they are denied all access to the running of the flow.
  2. Swimlane authentication: During the execution of the flow, the user may enter into a swimlane element (which is a group of elements that have a particular authentication configuration that differs from the flow. In this situation, the default behavior of our runtimes is that the user will not be excluded from the flow and can continue to get status updates and collaborate. However, the user cannot view any pages or take any actions.

The engine will provide you with the authorization context as part of the response. The authorization context can then be used against our authentication API to log the user in to the correct directory to get the required authentication token. The authorization context object will be returned for initialization, execution and join requests on the engine. We do not return the authorization context for "response" requests as part of an asynchronous execution.

Initializing A Flow Start your engines :)

Once you have the full Flow Identifier object, you can initialize your flow, ready to start executing! Initialization allows you to do a few things:

  1. Create inputs: Inputs are related to Values in a flow. If the author of the flow has created Values that have an access setting of "Input" or "Input/Output", you can pass information into those values at initialization. These might be things like the Id of an Account record in salesforce.com that the flow relates to, or the phone number of the current caller.
  2. Create annotations: An annotation is a piece of data you want to attach to your flow, but is not actually passed in as an input. Often developers want to store additional information in the flow state that helps with integration or more complex configuration. Think of annotations as session state that is read/write for the entire duration of the flow execution.
  3. Set the execution mode: The "mode" allows you to run the flow in a couple of different debug configurations. This allows you to step through the flow as a user would see it, but also step through the flow element by element (even logical elements). It's very powerful for debugging - honestly - it's fantastic when flows get a little more complicated!

Realtime Flows Collaborating!

In addition to the currently running user of your app, there may be others also making changes and performing operations on the same flow state. To manage this, we provide an API whose sole responsibility is to tell the player if any changes have happened outside the scope of this user. If a change has occurred then we recommend that the engine perform a "SYNC" invokeType request to get the most up-to-date information from the state. This API should be used for structural and navigation changes. Content synchronization will be done using a separate operational transform API that will be released when the page composition tool is completed.

Currently the player app needs to poll this API periodically to check for changes. The frequency of polling will depend on the type of application, though our default player implementation pings every 10 seconds. In future we will be providing this as a streaming API.

API Enumerations Helpful values

InvokeType Enumeration

The InvokeType enum contains the following values:

Value Description
FORWARD Tells the engine that you wish to move forward. This doesn't necessarily mean the engine will move forward if the request fails on page validation OR the author of the flow has put rules against the element that have not evaluated to 'true'.
DONE This invoke type is returned by the engine if the flow has completed.
WAIT Tells the engine and the caller that the engine is waiting or the external plugin is waiting. As part of the plugin framework a "WAIT" request allows you to send status messages to the user without progressing the flow. When calling the engine from a plugin, if the engine responds with a "WAIT", it means the engine is likely waiting on another event or business rule to fire.
STATUS Tells the engine that you wish to get a status update on the flow. This will send back up-to-date page metadata and any new status messages.
SYNC Tells the engine that you wish to synchronize the data held in the player with that held on the service. This will send back up-to-date page metadata and any new status messages.
SUCCESS When making asynchronous requests against the engine, a "SUCCESS" response tells the caller that the request was accepted and the caller therefore does not need to continue to make requests against the engine.
TOKEN_COMPLETED When making asynchronous requests against the engine, a "TOKEN_COMPLETED" response tells the caller that the request is no longer needed or valid and the caller therefore does not need to continue to make requests against the engine. This can happen if the user has send messages out to multiple systems and the rules to continue forward in the execution of the flow were satisfied by another system response.

PageActionBinding Enumeration

The PageActionBinding enum contains the following values:

Value Description
SAVE The engine should apply all values collected to the state.
NO_SAVE The engine should ignore all values collected and not apply anything to the state except the outcome action.

ContainerType Enumeration

The ContainerType enum contains the following values:

Value Description
VERTICAL_FLOW Child page components and containers should be organized vertically.
HORIZONTAL_FLOW Child page components and containers should be organized horizontally.
INLINE_FLOW Child page components and containers should be organized inline (meaning they should appear side-by-side and wrap if there's not enough space horizontally.
GROUP Child page components and containers should be organized as a group. The default runtime interprets a "GROUP" as tabs.

ComponentType Enumeration

The ComponentType enum contains the following values:

Value Description
CONTENT The component collects HTML5 formatted content.
INPUT The component is a field input collecting anything from dates to passwords to text.
TEXTAREA The component supports unformatted, multi-line content.
SELECT The component allows the user to select from a list of options provided by the supplied object data or object data request.
CHECKBOX The component is a simple Yes/No checkbox.
TABLE The component allows the user to select, edit and delete entries in a list of options provided by the supplied object data or object data request.
PRESENTATION The component presents HTML5 formatted content to the user.
TAG The component is unknown and the player should refer to the tag information to determine how to display the provided value or object data.
IMAGE The component is an image reference.

ContentType Enumeration

The ContentType enum contains the following values:

Value Typed Description
ContentString No A Java java.lang.String or C# System.String value.
ContentNumber No A Java java.lang.Double or C# System.Double value.
ContentBoolean No A Java java.lang.Boolean or C# System.Boolean value.
ContentPassword No A Java java.lang.String or C# System.String value. If you indicate a value is a password, we use additional value encryption on storage.
ContentDateTime No A Java java.lang.DataTime or C# System.DateTime. Using the REST API, the value should be provided in JavaScript 1.8.5/ECMAScript 5th Edition format.
ContentContent No A Java java.lang.String or C# System.String value. If you indicate a value is "Content", we typically expect that the content holds HTML formatted String content. The engine will also parse the contents of this String with any value references.
ContentObject Yes The value is Object Data based on the ManyWho format. This allows you to pass complex data into ManyWho for processing, including child objects and lists. Any Object Data being passed into the system also needs to have a Type as defined by your Type Elements.
ContentList Yes The value is a list of Object Data based on the ManyWho format. This allows you to pass lists of complex data into ManyWho for processing (of the same Type), including child objects and lists. Any Object Data being passed into the system also needs to have a Type as defined by your Type Elements.

Mode Enumeration

The Mode enum contains the following values:

Value Description
null or empty String Run the flow in full release mode.
DEBUG Run the flow with debug information switched on. This will populate the state values in the response so you can see what is ready to be committed and what values have already been committed to the state.
DEBUG_STEPTHROUGH Run the flow with debug information switched on, but also step through element by element. This allows you to see exactly what's happening as your flow executes as you are given the opportunity to see steps that are normally not shown to the user and track the path of execution and the state information as the engine executes.
